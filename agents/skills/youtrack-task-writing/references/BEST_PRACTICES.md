# YouTrack Task Best Practices

## Default Template

```md
Title: <specific outcome or problem>

Context
- Why this task exists now
- Who is affected

Problem / Goal
- What must change

Scope
- Included behavior or surfaces

Out of scope
- Explicit exclusions

Acceptance criteria
- Observable result 1
- Observable result 2
- Edge-case behavior if relevant

Dependencies / Links
- Related issue IDs, incidents, PRs, docs

Notes / Risks
- Rollout constraints, migration notes, open questions
```

## Best Practice Checklist

- Title is concrete and searchable.
- Description explains why the work exists.
- Scope is bounded.
- Acceptance criteria are testable.
- Unknown field values are verified through YouTrack MCP.
- Dependencies are linked instead of being hidden in prose.
- Out-of-scope items are explicit when confusion is likely.
- The issue can be estimated without a follow-up meeting.

## Bad vs Good Examples

### Example 1: Bug

Bad:

```md
Title: Fix login bug

Users have some login issues. Investigate and fix.
```

Good:

```md
Title: Prevent SSO login failure when email contains uppercase letters

Context
- Support reported repeated login failures for enterprise users authenticating via SSO.
- Failures began after the new identity provider sync was enabled.

Problem / Goal
- Users whose email contains uppercase characters cannot complete SSO login.
- Normalize email matching so login succeeds regardless of email casing.

Scope
- SSO login path
- Identity lookup by email
- Error handling for unmatched identities

Out of scope
- Changes to the identity provider configuration
- Full auth flow refactor

Acceptance criteria
- A user with `User.Name@Company.com` can log in through SSO if the stored account is `user.name@company.com`.
- Failed login telemetry preserves the real failure reason.
- Existing non-SSO login behavior remains unchanged.

Dependencies / Links
- Incident: AUTH-481
- Related issue: IAM-223

Notes / Risks
- Verify whether audit logs or rate-limit logic depend on original email casing.
```

Why this is better:

- Names the failing scenario.
- States expected behavior.
- Prevents accidental scope expansion.
- Gives acceptance criteria that QA can validate.

### Example 2: Feature

Bad:

```md
Title: Export improvements

Need to improve export so users can filter data better.
```

Good:

```md
Title: Add date range filter to CSV export for audit logs

Context
- Customers exporting audit logs for compliance review currently download the full dataset and filter it manually.

Problem / Goal
- Allow users to export audit logs for a selected date range.

Scope
- Audit log export UI
- Export API request parameters
- Validation for invalid date ranges

Out of scope
- New export formats
- Saved filter presets

Acceptance criteria
- Users can select start and end dates before export.
- Exported CSV contains only records inside the selected range.
- If start date is after end date, the UI blocks export and shows a validation message.
- If no date range is selected, current export behavior remains unchanged.

Dependencies / Links
- Design spec: verify via YouTrack MCP or linked design issue

Notes / Risks
- Confirm whether exports use UTC or workspace timezone.
```

### Example 3: Spike

Bad:

```md
Title: Research queues

Look into queue options for background jobs.
```

Good:

```md
Title: Evaluate queueing options for high-volume background job processing

Context
- Current in-process worker model is struggling with retry storms during peak imports.

Problem / Goal
- Compare viable queueing approaches and recommend one for the next implementation phase.

Scope
- Assess operational complexity, retry model, observability, throughput, and failure handling.
- Compare at least the current approach, managed queue, and self-hosted broker options.

Out of scope
- Production implementation
- Full migration plan

Acceptance criteria
- The issue ends with a short recommendation document.
- The document compares options against explicit criteria.
- The recommendation includes tradeoffs, risks, and a proposed next step.

Dependencies / Links
- Postmortem: OPS-912

Notes / Risks
- Keep analysis constrained to workloads generated by import and webhook processing.
```

## Field Hygiene

When working against a real project:

- verify actual field names and enum values through YouTrack MCP
- use the project's real issue type instead of generic guesses
- confirm whether the project expects story points, estimation, subsystem, sprint, fix version, or tags
- check existing linked issues before creating duplicates

## Rewrite Heuristics

If the input is vague, improve it in this order:

1. Replace generic verbs with the failing scenario or target outcome.
2. Extract or infer the business context from the request.
3. Separate required behavior from implementation ideas.
4. Convert "done means fixed" into explicit acceptance criteria.
5. Mark unknown facts for MCP verification instead of inventing them.
